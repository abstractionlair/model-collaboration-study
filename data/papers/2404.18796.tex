\begin{abstract}
\end{abstract}
As Large Language Models (LLMs) have become more advanced, they have outpaced our abilities to accurately evaluate their quality. Not only is finding data to adequately probe particular model properties difficult, but evaluating the correctness of a model's free-form generation alone is a challenge. To address this, many evaluations now rely on using LLMs themselves as judges to score the quality of outputs from other LLMs. Evaluations most commonly use a single large model like GPT-4. While this method has grown in popularity, it is costly, has been shown to introduce intra-model bias, and in this work, we find that very large models are often unnecessary. We propose instead to evaluate models using a \method (\acronymnospace). Across three distinct judge settings and spanning six different datasets, we find that using a \acronym composed of a larger number of smaller models outperforms a single large judge, exhibits less intra-model bias due to its composition of disjoint model families, and does so while being over seven times less expensive. 

\section{Introduction}
Evaluating generative language models is a challenging task: not only is it difficult to find meaningful data to test the models, but evaluating the correctness of a generated response is itself a challenge. Multiple choice datasets like MMLU~\cite{hendrycks2020measuring} have become popular in part by side-stepping the difficulty of evaluating generations. However, multiple-choice questions are in many ways probing a different property than that of a free-form generative task, which is oftentimes closer to the downstream use-case.

Many automatic metrics have been used across various tasks such as BLEU in machine translation~\citep{papineni2002bleu}, ROUGE for summarization~\citep{lin-2004-rouge}, and heuristic string match methods, such as normalized exact match (EM) and token level F1 for question answering~\cite{rajpurkar2016squad}. However, these simplistic methods commonly fail to analyze the intended property of interest. QA metrics, for example, invariably lead to both false positive failures (e.g. superfluous token overlap) and more commonly false negatives due to an incomplete set of gold reference answers (e.g. date format differences\footnote{We found that EM unjustly penalized Command models for a tendency to write in Canadian or British English as QA dataset annotations typically format dates in American MM-DD-YYYY format.}, inclusion of middle initial in person's name, etc.).

More recent methods have attempted to address these issues by instead using trained or prompted models as evaluators~\cite{sellam-etal-2020-bleurt, zheng2024judging, li2024leveraging, kocmi-federmann-2023-large, shen-etal-2023-large, li2023prd}. Prior work has shown that model-based scoring methods often correlate better with human judgements than heuristic metrics like EM~\cite{bohnet2022attributed,zheng2024judging} and that strong evaluator models generalize well across different tasks~\cite{huang2024empirical}. 

Unfortunately, while the use of LLMs like GPT-4 as evaluators has become increasingly common, it has also been observed that evaluator models tend to have their own biases; often recognizing and preferring their own outputs over those of other models~\cite{Panickssery2024LLMEval}. Additionally, it is most common to use the largest, most universally capable models as evaluators, which is both slow and costly, limiting applicability and access. 

In this paper, we perform experiments across three settings (single-hop QA, multi-hop QA, and Chatbot Arena), spanning six datasets, and make the following contributions:
\begin{enumerate}
\item We propose to evaluate LLM generations using a \method (\acronymnospace) drawn from different model families rather than a single large judge (Section \ref{sec:method}).
\item We show that using an instantiation of \acronym correlates better with human judgements compared to a single large judge (GPT-4), while being over seven times cheaper (Sections \ref{sec:results_kappa} and \ref{sec:result_arena}).
\item In some scenarios, GPT-4 is a relatively weak judge, exhibiting high variance with minor changes to the prompt (Section \ref{sec:prompt_variance}).
\item Intra-model scoring bias is reduced by pooling judgements across a panel of heterogeneous evaluator models (Section \ref{sec:result_bias}).
\end{enumerate}



\section{Methods \label{sec:method}}
\subsection{Background: LLM as a Judge \label{sec:judge_background}}
A judge evaluator model $J$ is used to score the output $a$ from a test model $A$. 

\noindent\textbf{Single-point Scoring}
In some settings, $J$ is tasked with rating the quality of a single model output independently of any point of comparison~(e.g.~\citep{kocmi2023large}). $J_{\textup{prompt}}$ will often include natural language instructions on how the grading should be performed (i.e. what properties constitute a good or bad output). Aside from the prompt instructions, the rating is based solely on $J$'s internal model of what a quality output is. Here, $\textup{score}=J(a)$.

\noindent\textbf{Reference-based Scoring}
In other cases, the model is provided with some 'gold' reference $r$, which contains the information that should be included in $a$~(e.g.~\cite{zhu2023judgelm}). For example, in QA the reference would be the 'correct' answer to the question. In this case, $\textup{score}=J(a,r)$. This setting is explored in Sections \ref{sec:single_qa} and \ref{sec:mh_qa}.

\noindent\textbf{Pair-wise Scoring}
Another very common setting is in pair-wise scoring where the goal is to choose which of two outputs is better~(e.g.~\cite{zheng2024judging}). Given outputs $a$ and $b$ generated by two models $A$ and $B$, an evaluator $J$ compares them and generates a preference score over the outputs as $\textup{score}=J(a,b)$\footnote{To avoid ordering bias, it is common to test both $a-b$ and $b-a$ orderings as done in Section \ref{sec:arena_hard}}. The form of the score can vary based on the use-case, but it is often a three or five point scale such as $a>b$, $a\approx b$, $a<b$. This setting is used in Section \ref{sec:arena_hard}.

\subsection{\methodcaps}
The above settings assume that all scoring is performed by a single capable judge. However, as outlined earlier, one of the largest issues with relying on a single model $J$, such as GPT-4, is that it introduces intra-model bias. To address this we instead propose to score answer correctness based not on a single judge, but instead on a panel composed of multiple evaluator models. Similar pooling techniques are used to reduce variance in human annotations by normalizing out both natural variation in human judgements caused by their own subjective biases as well as human error~\cite{vorhees1998variation}. In prior relevant work, \citet{li2023prd} proposed an evaluation method with multiple judges to reduce bias though only looked at large models on pair-wise evaluations.

To calculate the \acronym score, each evaluator model independently scores a given model output just as they would in any of the scenarios outlined above. Those individual scores are then pooled together through a voting function\footnote{In this work we consider both max and average pool voting, though other functions could also work well.} such that the final score  $= f(j \in P: j(a))$ where $P$ is a panel composed of individual judges $j$ and $f$ is a voting function. 

\section{Experimental Settings}

\subsection{\acronym Composition and Voting}
In our experiments, We construct a \acronym from three models being drawn from three disparate model families (Command 
R, Haiku, and GPT-3.5). We consider two different voting functions for aggregating scores across the judges. For QA datasets, we use max voting, as all judgements are binary [correct, incorrect]. For Chatbot Arena we instead use average pooling because judgements are scores ranging from 1-5 and a three judge panel often does not produce a clear majority decision\footnote{We also experimented with max voting with average voting fallback which yielded similar results with an identical overall ranking.}. 

\subsection{Model Families}
\noindent\textbf{Command R Family:}
Command R (CMD-R, 35B) and Command R+ (CMD-R+, 103B) are open weight models created by Cohere\footnote{Release blog of Command R \url{https://cohere.com/blog/command-r} and Command R+ \url{https://cohere.com/blog/command-r-plus-microsoft-azure}}.
We consider Command R as one of the models in the \acronymnospace.

\noindent\textbf{GPT Family:}
GPT-3.5 and GPT-4~\cite{openai2024gpt4} are two of the most widely used models. GPT-4 is typically the default choice for models as judges. We use GPT-3.5 as a member of our \acronymnospace.

\noindent\textbf{Claude-3 Family:}
Models built by Anthropic include (in increasing size) Haiku, Sonnet, and Opus~\cite{anthropic2024sonnet}. We include Haiku in our \acronymnospace. 

\noindent\textbf{Mistral Family:}
Mistral Large (Mistral-LG) and Mistral Medium (Mistral-MD) are proprietary models created by Mistral\footnote{Release blog of Mistral Large: \url{https://mistral.ai/news/mistral-large/} and Mistral Medium: \url{https://mistral.ai/news/la-plateforme/}.}. We did not conduct experiments with Mistral judges but evaluate their generations as a point of comparison to have a model 'unaffiliated' with any judges.

\subsection{Single-hop Question Answering \label{sec:single_qa}}
Our question answering (QA) tasks are open-book settings where a model $m$ is given a question $q$ and must retrieve evidence $e$ from some retrieval system (such as the internet or dense index over wikipedia) and must generate an answer $g$ as $g=m(q, e)$. Each instance is accompanied by a human annotated reference answer. 

We run experiments on datasets from KILT~\cite{petroni2021kilt} versions of Natural Questions (NQ)~\cite{kwiatkowski-etal-2019-natural}, TriviaQA (TQA)~\cite{joshi-etal-2017-triviaqa}, and HotpotQA (HPQA)~\cite{yang-etal-2018-hotpotqa} (see Section \ref{sec:appendix_qa} for more details).


\subsection{Multi-hop Question Answering \label{sec:mh_qa}}
Multi-hop question answering is similar to the single-hop setting described in Section \ref{sec:single_qa}. However, the questions are designed such that models must perform multiple rounds of retrieval to answer sub-questions and collect sufficient evidence to ultimately answer the initial question. We perform experiments on two datasets: Bamboogle~\citep{press2023measuring} and HPQA. See Section \ref{sec:appendex_react} for additional details.

\subsection{Chatbot Arena Hard \label{sec:arena_hard}}
Chatbot Arena\footnote{\url{https://chat.lmsys.org/}} is a popular benchmark for evaluating LLM head-to-head performance. This crowd-sourced effort has users prompt a pair of anonymized LLMs and rate which output is better~\cite{zheng2024judging}. 

Recently,~\citet{arenahard2024} mined a 'hard' subset of 500 arena prompts and created an automatic evaluation where an LLM judge (GPT-4) would score a model output against a baseline generation (coming from a different version of GPT-4) in a head-to-head comparison. In their analysis, they determined that this evaluation correlated better with the human judgements from the original Chatbot Arena annotations when compared against other commonly used automated evaluation benchmarks. 

At the time of writing, the repository only contained generated answers from GPT-3.5 and GPT-4. We therefore generated new outputs for all remaining models and subsequently scored generated answers with the various judges using the original codebase from~\citet{arenahard2024}\footnote{\url{https://github.com/lm-sys/arena-hard}}. The raw scores can be found in Table \ref{tab:arena_hard_scores}. Following the original work, we treat Chatbot Arena crowdsourced annotations as ground truth for calculating correlation between evaluator models and human judgements.

\subsection{Prompting Judges}
As alluded to in Section \ref{sec:judge_background}, judge models need to be prompted in different ways depending on the particular task setup. Our QA experiments use reference-based scoring and our models prompts contain few-shot in-context examples of valid and invalid $q, a, r$ triples. We also experiment with variations on these prompts in Section \ref{sec:prompt_variance}. Further details on judge prompts may be found in the appendix.

As an additional point of reference in our QA experiments, we include the commonly used exact match (EM) metric judgements. We specifically use the 'containment' version of EM from prior work which is more amenable to LLM long-form generation~\citep{liu2024lost} and checks if a reference answer string appears within the generated model response (after normalization).

In our experiments on Chatbot Arena Hard, we follow the same pair-wise scoring setup of the original work. We do not modify the judge prompts in any way from the original implementation. 


\subsection{Human Judgements}
To gather human reference judgements, we utilized Cohere's internal highly-qualified annotation workforce. Annotators were shown a single anonymized model generated answer at a time along with the original question and reference answer. Annotators were asked to judge whether the reference answer is semantically contained inside the generated answer\footnote{By asking for answer correctness alone, we hope to minimize any potential annotator bias or preference towards a particular model's style.}.  See Section \ref{sec:appendix_annotation} for additional details 

\section{Results \label{sec:results}}

\subsection{Correlation to Human Judgements \label{sec:results_kappa}}
\noindent\textbf{Cohen's $\kappa$ Correlation}
Cohen's kappa measures inter-rater reliability, which quantifies the level of agreement between two or more raters or judges. The kappa statistic takes into account the possibility of agreement occurring by chance, making it a more robust measure than simple percent agreement.

The formula for Cohen's kappa is:
$$
\kappa = \frac{p_o - p_e}{1 - p_e}
$$

\noindent where $p_o$ is the observed agreement proportion and $p_e$ is the expected agreement proportion by chance. This results in a value of $\kappa$ ranging from -1 to 1. Opinions vary on how scores should be interpreted, but in general $\kappa > 0.8$ is considered a strong correlation and $\kappa > 0.6$ is a moderate correlation.

In Table \ref{tab:judge_model_performance} we can see how the ratings from different evaluator judges, on different single-hop QA datasets from KILT, correlate with human judgements as measured by $\kappa$. We see that overall, \acronym has the strongest correlation across various tasks, while GPT-4 is one of the weaker evaluators on this particular task setup (See \ref{sec:prompt_variance} for further analysis).

\subsection{Judgement Variance by Prompt Changes \label{sec:prompt_variance}}
Based on the observation that GPT-4 was the weakest judge model on our KILT evaluations, we investigated how the model reacts to modifications to its prompt. GPT-4 is the most powerful judge model we tested, yet it performed worse than less capable models on what is essentially a fuzzy string matching exercise. We hypothesize that may be because GPT-4 is over-reasoning and injecting too much background knowledge into determining the correctness of an answer rather than simply aligning the gold reference with the generation. In Table \ref{tab:prompt_variants}, we can see how the correlation between GPT-4 and human annotators varies as the prompt changes. In all cases, having in-context examples improves the performance over zero-shot and the most effective strategy is an explicit instruction to the model not to 'overthink' and not to concern itself with the wider factuality of the answers with respect to the outside world.\footnote{the 'don't overthink' prompt had a negligible positive effect on Haiku, minor negative effect to Command R and a severe negative impact on GPT-3.5.} These changes bring the agreement level for GPT-4 up to the level of GPT-3.5 when using our few-shot standard prompt, though still below Command-R, Haiku, and \acronymnospace.

\subsection{Judge Bias and Consistency \label{sec:result_bias}}
One of the biggest motivators for replacing a single large judge with a panel of heterogeneous models is to reduce bias in evaluation. To analyze to what extent this was the case, we compared the delta in absolute accuracy score for our individual judges and \acronym relative to scores by human annotators across our multi-hop datasets. Figures \ref{fig:hpqa_multi_scatter} and \ref{fig:bamboogle_scatter} show results on HotPotQA and Bamboogle. We can see how the different judges score different models and how far those predictions deviate from human annotator decisions (the dotted line at 0). We observe that overall, \acronym has the smallest spread in scores, with a standard deviation of 2.2, compared to EM and individual judges. GPT-3.5 has the highest spread, with a standard deviation of 6.1. We also see in Figure \ref{fig:bamboogle_scatter} that the highest positive delta for each individual model being scored occurs when it is judged by itself.

\subsection{Cost and Latency}
At the time of writing, the cost of running our specific instance of \acronym is \$1.25/input\footnote{per million tokens} + \$4.25/output, whereas the cost of running GPT-4 Turbo is \$10/input + \$30/output. Depending on the ratio of input-to-output tokens in a given task, running the entire three model \acronym is seven to eight times less expensive than running a single GPT-4 judge. 

We did not run explicit latency evaluations and many factors such as model choice, serving platform, and more can impact speed. In general though, running a collection of smaller models in parallel (as in \acronymnospace) is faster than a single big model.

\section{Conclusions and Limitations}
In this paper, we showed how a \methodcaps composed of smaller models is not only an effective method for evaluating LLM performance, but also reduces intra-model bias, latency, and cost. The benefits of \acronym are bolstered by the finding that there is not a single 'best' judge across all settings, while \acronym performs well consistently.

In this work we investigated only three evaluator settings and a limited number of judges and panel compositions. While we showed that \acronym is an effective alternative to a single large model in these settings, further work is needed to see how broadly applicable the method is, for example, in math or reasoning evaluations, where language models often struggle~\cite{zheng2024judging}. We also leave the task of 'panel selection', or identifying the best models to include in \acronym in terms of quality and cost, to future work.

\section{Appendix}
\subsection{Additional Experimental Details}
\subsubsection{Single-hop QA \label{sec:appendix_qa}}
\noindent\textbf{Datasets}
KILT~\cite{petroni2021kilt} is a popular benchmark that measures a model's open domain question-answering capabilities with 1) a retrieval step to retrieve relevant documents from a knowledge source and 2) an answer generation step taking into account the relevant documents from (1). We select three popular test sets from the KILT benchmark, Natural Questions (NQ)~\cite{kwiatkowski-etal-2019-natural}, TriviaQA (TQA)~\cite{joshi-etal-2017-triviaqa}, and HotpotQA (HPQA)~\cite{yang-etal-2018-hotpotqa} for evaluation. The Natural Questions test set consists of real user questions sampled from a popular web search engine. %that require external knowledge from Wikipedia to be answered.
TriviaQA is a test set with complex questions written by trivia enthusiasts. %, that require the combination of multiple evidences from Wikipedia documents to be answered.
HotpotQA is a multi-hop question answering test set where questions require finding and reasoning over multiple supporting documents to answer. 
For all three test sets, the knowledge source is the corpus of Wikipedia documents form the KILT benchmark~\cite{petroni2021kilt}.

\paragraph{KILT Experimental Details:} For KILT, we use the DPR-formatted 100-word chunked KILT wikipedia dump~\cite{Karpukhin2020DensePR,petroni2021kilt}.\footnote{The pre-chunked dump can be downloaded here: \url{https://github.com/facebookresearch/DPR/issues/186\#issuecomment-923268677}}
Then, for all questions, we retrieve and cache the top-10 snippets from the dump, using the GTE-large dense retrieval model~\citep{Wang2023ImprovingTE}, ready to be used to generate answers from different models.

\paragraph{Generating answers for judgement:} To generate answers from all considered models, we ask the question as an unmodified chat message and pass in the snippets using the dedicated \texttt{documents} parameter of the model family's API where available (Command-R Family)\footnote{The prompt template that gets used when using this parameter is described at \url{https://huggingface.co/CohereForAI/c4ai-command-r-v01} with additional details at \url{https://docs.cohere.com/docs/prompting-command-r}}. 
For model families that do no have a specific \texttt{documents} api parameter (GPT, we instead adopt the question answering prompt template used in~\citet{liu2024lost}, which is shown in table~\ref{tab:litm_prompt}.

\paragraph{Judge Prompts} We use the same prompt-template structure for KILT judgements as for Multihop judgements. We initially observed relatively-lower levels of human-agreement when using the multihop prompt verbatim applied to KILT results experiments. 
We hypothesise this was due to mismatch in model-generated answer styles in the few-shot examples.
We therefore create new judgement prompts the following procedure for each of the KILT datasets, randomly sampling a set of labelled fewshot examples from our pool human-labelled examples according to the following constraints:
\begin{itemize}
    \item at least 1 annotated positive example from each model family's generations
    \item at least 1 annotated negative example from each model family's generations
    \item at least 1 example where the human disagrees with keyword-based exact match score.
\end{itemize}

We sweep over \texttt{n\_shots} in $\{8,10,16\}$, picking the prompt that has strongest agreement on average across all judges for a small held-out portion of human labels. The resuling judgement prompts used for KILT in our experiments in the main paper are listed in tables \ref{tab:hotpotqa_kilt_prompt}, \ref{tab:nq_kilt_prompt} and \ref{tab:triviaqa_kilt_prompt}.

\paragraph{GPT-4 judge prompt ablation} Here we include some additional details on prompt ablation for KILT NaturalQuestions mentioned in section \ref{sec:prompt_variance}:
\begin{itemize}
    \item \textbf{Zero-shot}: the natural language instruction is supplied as system call. We then directly ask the model to judge with no labelled examples
    \item \textbf{Few-shot Standard}: This is the prompt in table \ref{tab:nq_kilt_prompt} which is used for all other model judges.
    \item \textbf{No instruction Line}: Here we remove the natural language instruction from the Few-shot standard prompt, on the hypothesis the instruction is confusing the model. This hypothesis turns out to be false as agreement actually drops. (-0.03 $\Delta \kappa$)
    \item \textbf{Move instruction Line}: Here we modify the standard fewshot prompt by moving the instruction line into a separate system call. This results in a small improvement (+0.01 $\Delta \kappa$)
    \item \textbf{Chat-formatted shots}: Here we modify the standard fewshot prompt by formatting each fewshot example as a conversational turn between the user and the assistant. This reduces performance (-0.07 $\Delta \kappa$)
    \item \textbf{`don't overthink'}: Here, we replace the instruction line from the standard prompt's \texttt{You will be given a Question and a Provided Answer. Judge whether the Provided Answer is correct by comparing it to the Reference Answer. Differently formatted dates, people with missing middle names, and alternative spellings should all be considered the same. If the Provided Answer is correct say exactly "True", otherwise say "False".} to a wording which is intended to encourage the model to perform a simpler function and not incorporate external knowledge: \texttt{ "You are judging whether a model has generated a correct answer to a question. Study the examples the user gives you as they will be very informative for how to do the task. The Reference Answers you get will be short. An model's answer will be longer, and can be considered correct if it contains the semantic content of short reference answer somewhere within it. Don't worry about factuality with respect to the real world, just judge the example based on what you see. No need to overthink this task, it really comes down to just soft matching.} This improves results for GPT-4 by +0.07 $\Delta \kappa$. Additional small surface level changes and moving the instruction to a system call lead to an additional +0.03 $\Delta \kappa$. The final optimized prompt for GPT-4 can be found in table \ref{tab:nq_kilt_prompt_gpt4_opt}.
\end{itemize}
